## 0. Самая большая картинка

Мы делаем замену Windows-программе `MultiCam.exe`, но на Linux (для Jetson на дроне).
Вся работа делится на **2 части**:

1. **Получить картинку с камеры** (слой `camera/`).
2. **Превратить картинку в полезные цифры о здоровье растений** (слой `processing/`).

```
            ┌─────────────┐     ┌───────────────────────────────────────────┐
КАМЕРА  →   │  camera/    │  →  │              processing/                  │  →  ЦИФРЫ + КАРТЫ
(железо)    │ (берём кадр)│     │ режем → совмещаем → отражение → индексы    │     (NDVI и т.д.)
            └─────────────┘     └───────────────────────────────────────────┘
```

### 🧒 Объясни как пятилетнему
Камера — это **фотоаппарат, который видит больше цветов, чем человек** (в том числе
«невидимый» свет). Наша программа берёт его фотку и считает, **здоровое растение или
болеет**, ещё до того как это станет видно глазом. Левая часть кода — «сделать фото»,
правая — «понять, что на фото».

---

## 1. Что за камера и что за данные

**Камера:** `BigEye4200KME` — один чёрно-белый сенсор **2048×2048 пикселей**, USB 3.0.

Хитрость: перед сенсором стоит сетка из маленьких объективов с разными
**светофильтрами**. Поэтому на одном большом кадре получается **сетка маленьких
картинок (тайлов)**, и каждая снята «в своём цвете» (длине волны).

Мы работаем с **12 длинами волн** (нанометры):

```
400  450  500  550  600  650  700  750  800  850  900  950
синий ────────── зелёный ── красный ── «красный край» ── ближний ИК (невидимый)
```

Эти числа заданы в коде один раз:

```10:14:linux/multicam/__init__.py
WAVELENGTHS = (400, 450, 500, 550, 600, 650, 700, 750, 800, 850, 900, 950)
"""Центральные длины волн 12 спектральных каналов, нм (порядок = порядок матриц в calibration*.txt)."""

ANCHOR_WAVELENGTH = 800
"""Опорный канал: в calibration*.txt ему соответствует единичная матрица (нулевой сдвиг)."""
```

- **Откуда данные:** из мануала (`Manual.pdf`) и калибровочных файлов.
- **Что дают:** для каждого пикселя мы знаем, как сильно он отражает свет в 12 разных
  «цветах». Это и есть «спектр».
- **Зачем:** разные вещества в листе (хлорофилл, вода, каротиноиды) по-разному отражают
  разный свет. По спектру можно «на расстоянии» понять биохимию растения.

### 🧒 Объясни как пятилетнему
Представь 12 разноцветных фонариков. Мы светим каждым на листик и записываем, сколько
света он отразил. Здоровый зелёный лист «зеркалит» невидимый свет очень сильно, а
больной — слабее. Камера делает это сразу всеми фонариками за один щелчок.

## 2. Полный путь данных (pipeline)

```
RAW-кадр 2048×2048
   │  split_channels()      ← режем на 12 тайлов (channels.py)
   ▼
12 «цветных» картинок {400:..., 450:..., ...}
   │  register_channels()   ← сдвигаем, чтобы совпали (registration.py + calibration.py)
   ▼
12 совмещённых картинок
   │  to_reflectance()      ← делим объект на эталон (reflectance.py)
   ▼
Отражение R (0..1) для каждого канала
   │  mean_spectrum()       ← усредняем по выбранной области
   ▼
Спектр {400:0.80, 450:0.35, ...}
   │  indices.compute_all() ← считаем NDVI, CIre... (indices.py)
   ▼
Цифры здоровья растения + карты
```

Этот путь собран в одном месте — в `pipeline.py` (разберём ниже).

## 3. Разбор каждого модуля

### 3.1 `io/calibration.py` — чтение калибровок

**Что делает:** читает файлы `calibration05.txt` / `calibration10.txt` /
`calibration15.txt` (цифры в имени = дистанция съёмки 0.5 / 1.0 / 1.5 метра).

**Что внутри файла:** 12 матриц 3×3 подряд. Каждая матрица — это «сдвиг» одного канала:

```
1 0 -22.636      ← dx = -22.636 (сдвиг по горизонтали)
0 1 -18.231      ← dy = -18.231 (сдвиг по вертикали)
0 0 1
```

**Откуда данные:** файлы поставлялись вместе с Windows-программой (заводская калибровка).

**Что дают:** на сколько пикселей надо подвинуть каждый «цветной» тайл, чтобы все 12
картинок идеально наложились друг на друга.

**Зачем:** маленькие объективы стоят в разных местах сетки, поэтому смотрят чуть-чуть
под разными углами. Без выравнивания «красный» лист и «синий» лист будут смещены, и
расчёты испортятся.

Ключевая функция — достать сдвиг для канала:

```41:44:linux/multicam/io/calibration.py
    def shift_for(self, channel_index: int) -> tuple[float, float]:
        """Возвращает (dx, dy) в пикселях для канала по индексу."""
        m = self.matrices[channel_index]
        return float(m[0, 2]), float(m[1, 2])
```

Важная деталь: у канала **800 нм** сдвиг нулевой — это «опорный» канал, к нему
подгоняют все остальные.

#### 🧒 Объясни как пятилетнему
Представь 12 листов кальки с рисунками, которые надо наложить точно один на другой,
но их слегка сдвинули. В файле калибровки записано: «этот листик подвинь на 22 мм влево
и 18 мм вверх». Один листик (800) лежит правильно, его не трогаем, а остальные двигаем
к нему.

### 3.3 `processing/channels.py` — разрезка кадра на каналы

**Что делает:** берёт большой кадр 2048×2048 и режет на сетку **4×4 = 16 тайлов** по
512×512. Из них 12 — наши спектральные каналы.

**Что даёт:** словарь `{длина_волны: маленькая картинка}`, например `{400: <512×512>, ...}`.

Геометрия тайла считается простой арифметикой:

```55:61:linux/multicam/processing/channels.py
    def tile_rect(self, tile_index: int, frame_h: int, frame_w: int) -> tuple[int, int, int, int]:
        """Возвращает (y0, y1, x0, x1) тайла в кадре."""
        th = frame_h // self.grid_rows
        tw = frame_w // self.grid_cols
        r = tile_index // self.grid_cols
        c = tile_index % self.grid_cols
        return r * th, (r + 1) * th, c * tw, (c + 1) * tw
```

> ⚠️ **Важно и пока не точно:** какой именно тайл соответствует какой длине волны —
> сейчас **заглушка** (берём по порядку). Это надо сверить с реальными данными. Поэтому
> карта вынесена в отдельный файл `config/channels.yaml`, чтобы поправить без изменения кода.

**Зачем:** дальше каждую длину волны обрабатываем отдельно.

Ещё есть `to_gray()` — если кадр пришёл как цветной BMP (3 одинаковых канала), берём
один, потому что сенсор всё равно чёрно-белый.

#### 🧒 Объясни как пятилетнему
Большая фотка — это как **лист с 16 марками** (4 ряда по 4). Мы аккуратно разрезаем его
ножницами на отдельные марки. Каждая марка — это «фото в одном цвете». Пока мы не на
100% уверены, какая марка какого цвета — это запишем отдельно и потом уточним.

### 3.4 `processing/registration.py` — совмещение каналов

**Что делает:** двигает каждый тайл на его `(dx, dy)` из калибровки, чтобы все 12
наложились ровно.

**Откуда данные:** сдвиги — из `calibration.py`.

Если установлен OpenCV — двигаем точно (с дробными пикселями), иначе запасной вариант
через numpy (целые пиксели):


**Зачем:** чтобы пиксель «лист» в канале 650 нм и пиксель «лист» в канале 850 нм
указывали на **одну и ту же точку растения**. Иначе индексы посчитаются по разным местам.

#### 🧒 Объясни как пятилетнему
Берём наши разрезанные марки-кальки и **сдвигаем каждую**, чтобы рисунки идеально легли
друг на друга. Теперь, если ткнуть пальцем в точку, на всех 12 картинках под пальцем —
один и тот же кусочек листа.

### 3.5 `processing/reflectance.py` — коэффициент отражения

**Что делает:** превращает «сырую яркость» в **коэффициент отражения** по формуле:

```
отражение = (объект − тёмное) / (эталон − тёмное)
```

- **объект** — кадр с растением;
- **эталон (reference)** — кадр белого листа/эталона при тех же настройках;
- **тёмное (dark)** — кадр с закрытым объективом (необязательно).

**Откуда данные:** объект и эталон снимаем камерой; для теста используем один и тот же BMP.

**Что даёт:** число от 0 до ~1 (у нас обрезано до 2 на всякий случай) — доля
отражённого света, **не зависящая от яркости лампы/солнца**.

```31:36:linux/multicam/processing/reflectance.py
        denom = np.where(np.abs(ref_f) > _EPS, ref_f, _EPS)
        refl = obj_f / denom
        if clip:
            refl = np.clip(refl, 0.0, 2.0)
        out[wl] = refl
```

Ещё есть `mean_spectrum()` — усредняет отражение по выбранной прямоугольной области
(ROI), чтобы получить один «средний спектр» вместо целой картинки.

**Зачем:** если просто смотреть на яркость, то в солнечный день всё ярче, в пасмурный
темнее, и сравнивать нельзя. А отражение — это «процент», он стабильный.

#### 🧒 Объясни как пятилетнему
Представь, что ты сравниваешь, насколько листик «зеркальный». Чтобы свет лампы не
обманывал, рядом кладём **белый лист-эталон** (он отражает почти всё). Делим «сколько
отразил листик» на «сколько отразил белый лист» — получаем честный процент. В темноте
или на ярком солнце процент будет почти одинаковый.

### 3.6 `processing/indices.py` — вегетационные индексы

**Что делает:** считает 6 «индексов здоровья» из спектра. Формулы взяты из мануала и
**сверены с реальным выводом штатной программы — совпадают до 5-го знака**.

```37:58:linux/multicam/processing/indices.py
def ndvi(s: Spectrum):
    return _safe_div(s[850] - s[650], s[850] + s[650])


def cire(s: Spectrum):
    return _safe_div(s[700], s[750])


def mcari(s: Spectrum):
    return ((s[750] - s[700]) - 0.2 * (s[750] - s[550])) * _safe_div(s[750], s[700])


def ndvire(s: Spectrum):
    return _safe_div(s[750] - s[700], s[750] + s[700])


def car(s: Spectrum):
    return (_safe_div(1.0, s[500]) - _safe_div(1.0, s[550])) * s[800]


def pssrc(s: Spectrum):
    return _safe_div(s[800], s[450])
```

**Что означает каждый индекс (агрономический смысл):**

| Индекс | Что показывает | Простыми словами |
|---|---|---|
| **NDVI** | общее «здоровье»/количество зелёной массы | насколько густая и живая зелень |
| **CIre** | хлорофилл (по «красному краю») | сколько в листе «зелёного двигателя» фотосинтеза |
| **MCARI** | хлорофилл с поправкой на фон | то же, но точнее, без влияния почвы |
| **NDVI700 (NDVIre)** | индекс листовой поверхности (LAI) | насколько много слоёв листьев |
| **CAR** | каротиноиды | пигменты стресса/старения |
| **PSSRc** | фотосинтетическая активность | насколько активно растение «работает» |

Хитрость в коде: одни и те же формулы работают **и для одного числа (средний спектр),
и для целой картинки (карта индекса)**, потому что `_safe_div` умеет делить и числа, и
массивы, и защищает от деления на ноль.

**Зачем:** это финальный полезный результат — цифры, по которым агроном (или нейросеть)
решает, где поле болеет.

#### 🧒 Объясни как пятилетнему
Из 12 «цветных» чисел мы по простым формулам делаем «оценки», как в школе. NDVI — это
как «общая оценка здоровья листа»: близко к 1 — отличник (сочная зелень), близко к 0 —
двоечник (сухой/больной). Остальные индексы — оценки по отдельным «предметам»
(хлорофилл, пигменты и т.д.).

### 3.7 `processing/custom_index.py` — свои формулы безопасно

**Что делает:** позволяет пользователю ввести свою формулу, например `x650 / x900`
(где `x650` = отражение на 650 нм), и безопасно её вычислить.

**Как:** разбирает выражение через `ast` (синтаксическое дерево) и разрешает **только**
`+ - * /`, скобки, числа и переменные вида `xДлинаВолны`. Никакой «чужой код» выполнить
нельзя.

**Зачем:** учёные постоянно придумывают новые индексы. Чтобы не лезть в код каждый раз,
дали безопасное «поле для формулы», как в штатной программе.

Защита проверена тестом: попытка `__import__('os').system(...)` отклоняется.

#### 🧒 Объясни как пятилетнему
Это как **калькулятор, который понимает только цифры и плюс-минус**. Если кто-то
попробует написать «злую команду» вместо формулы — калькулятор скажет «не понимаю» и
ничего плохого не сделает.

### 3.9 Слой камеры `camera/`

Здесь всё устроено так, чтобы код **не зависел от наличия реальной камеры**.

- **`base.py`** — описывает «договор»: любая камера умеет `open / set_exposure /
  grab / close`. Это как розетка-стандарт: неважно, что воткнуто, штекер один.
- **`mock_backend.py`** — «ненастоящая» камера: вместо железа отдаёт кадры из файлов.
  Нужна, чтобы тестировать в виртуалке без камеры.
- **`nncam_sdk.py`** — низкоуровневый «переводчик» (ctypes) к библиотеке камеры
  `libnncam.so` / `nncam.dll`. Описывает структуры и функции SDK.
- **`nncam_backend.py`** — настоящая камера: повторяет логику штатной программы
  (`Enum → Open → StartPullMode → PullImage`).
- **`__init__.py`** → функция `open_camera(backend=...)` выбирает: `mock`, `nncam` или
  `auto` (сначала пробует реальную, если нет — mock).

**Почему `nncam` = `toupcam`:** внутри `nncam.dll` все функции называются `Toupcam_*`.
Это переименованный SDK фирмы **ToupTek**, у которого есть **официальная версия под
Linux ARM64** (как раз для Jetson). Поэтому железную часть **не пришлось взламывать** —
используем готовую библиотеку вендора.

```108:131:linux/multicam/camera/nncam_sdk.py
def load_library(explicit_path: str | None = None) -> C.CDLL:
    """Загружает разделяемую библиотеку SDK."""
    errors: list[str] = []
    names = [explicit_path] if explicit_path else _candidate_lib_names()
    for name in names:
        if not name:
            continue
        try:
            return C.CDLL(name)
        except OSError as exc:
            errors.append(f"{name}: {exc}")
```

#### 🧒 Объясни как пятилетнему
Мы сделали **переходник для камеры**. Есть «понарошку-камера» (показывает заранее
сохранённые фотки — чтобы тренироваться без настоящей) и «настоящая камера». Для кода
они выглядят одинаково, поэтому можно сначала всё проверить на «понарошку», а потом
просто воткнуть настоящую — и ничего переписывать не надо.

### 3.10 `cli.py` — пульт управления

**Что делает:** даёт команды в терминале:

| Команда | Что делает | Нужна ли камера |
|---|---|---|
| `selftest` | проверяет формулы по эталону | нет |
| `process` | прогоняет фото через весь pipeline | нет |
| `probe` | ищет камеру через SDK | да |
| `capture` | снимает N кадров в файлы | да (или mock) |

## 4. Какие данные мы реально получаем и что они значат

| Данные | Откуда | Что это | Зачем |
|---|---|---|---|
| RAW-кадр 2048×2048 | камера / BMP-файл | сырая чёрно-белая картинка с сеткой тайлов | исходник всего |
| Калибровки (dx, dy) | `calibration*.txt` | сдвиги каналов | выровнять каналы |
| Эталон (reference) | съёмка белого листа | «100% отражение» | убрать влияние освещения |
| Отражение R(λ) | расчёт | процент отражения на каждой длине волны | основа всех индексов |
| Спектр | усреднение по области | 12 чисел | «портрет» участка листа |
| Индексы | формулы | 6+ чисел о здоровье | решение: болеет/здоров |
| Карты индексов | формулы по всему кадру | картинки NDVI и т.д. | где именно на поле проблема |

## 5. Что ещё НЕ доделано (честно)

1. **Карта «тайл → длина волны»** (`config/channels.yaml`) — заглушка, надо сверить с
   реальными снимками. До этого индексы на реальных кадрах могут быть неверны, хотя
   сами формулы корректны.
2. **Калибровки лабораторные** (0.5/1/1.5 м). Для дрона (высота десятки метров) их
   нужно пересчитать — об этом прямо написано в мануале.
3. **Геотеги** — модуля пока нет. План: брать координаты от полётного контроллера
   дрона (MAVLink) и вшивать в EXIF/GeoTIFF при сохранении кадра.
4. **Связь с нейросетью** — следующий этап после получения корректных снимков с гео.

### 🧒 Объясни как пятилетнему
Машинка уже **ездит по двору** (считает всё правильно на тестовой фотке). Осталось:
точно подписать, где какой цвет; научить её ездить **высоко с дрона**, а не только на
столе; приклеить к каждой фотке **GPS-наклейку «где снято»**; и отдать фотки «умному
мозгу» (нейросети).

## 6. Короткая шпаргалка «что где лежит»

```
linux/
  multicam/
    __init__.py            — длины волн, версия
    io/
      calibration.py       — читать сдвиги каналов
      series.py            — читать настройки съёмки
    processing/
      channels.py          — резать кадр на 12 каналов
      registration.py      — выравнивать каналы
      reflectance.py       — считать отражение + средний спектр
      indices.py           — NDVI, CIre, MCARI, NDVIre, CAR, PSSRc
      custom_index.py      — безопасные пользовательские формулы
      pipeline.py          — «сделай всё»
    camera/
      base.py              — общий интерфейс камеры
      mock_backend.py      — камера-понарошку (файлы)
      nncam_sdk.py         — низкоуровневый мост к libnncam.so
      nncam_backend.py     — настоящая камера
    cli.py                 — команды терминала
  config/channels.yaml     — карта тайлов (уточняется)
  tests/                   — автотесты
  README.md, TESTING.md    — инструкции
```
